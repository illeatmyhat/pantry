# Adding a locale

A locale in pantry is **a BCP-47 row plus data files — never a code change to
prose.** The prompt, schema, validator, and emitter all derive from the locale
table, and every localized string lives in a reviewable data file (no foreign
strings hardcoded in scripts). This guide is the full path for a new target,
end to end.

> Read [DESIGN.md](../DESIGN.md) first — identity, the leaf/view law, the
> errand model, and the localization decisions are settled there. This is the
> *how*; DESIGN is the *why*.

## 1. What a locale is made of

Five data surfaces. Knowing which are **generated** (from the zip, never
committed), **sourced** (researched, committed), or **human** (hand-written,
wins over everything) tells you where the work is:

| Surface | Home | Kind | Effort |
|---|---|---|---|
| Locale row | `scripts/translate/locales.ts` | code (config, no l10n strings) | minutes |
| Errand sections | `l10n/vocabulary/<tag>.yaml` `sections:` | sourced (discovered from data) | a discovery pass + signage review |
| Store labels | `l10n/vocabulary/<tag>.yaml` `stores:` | sourced (3 values) | minutes |
| Nutrient names | `l10n/nutrients/<tag>.yaml` | sourced (national standard + INFOODS) | **the hard part — see §4** |
| Food baseline | `l10n/baseline/<slug>.yaml` (per-food) | generated by batch, then reviewed | the paid corpus run |
| Ground truth | `l10n/ground-truth/<tag>.yaml` | human, durable, wins | ongoing |

The en-US surfaces are special: en-US names *are* the USDA data copied
mechanically (food name = description; nutrient names = USDA names + FDA panel
wording), so they are **generated, never committed**. Every other locale
supplies its own, committed.

A consumer reaches all of this through two shipped artifacts per locale
package: the `sr/<slug>` views (food + nutrition) and `labels.js`
(`{ sections, stores, nutrients }`), resolved by `localizeErrand` and
`localizeNutrients` in the toolkit.

## 2. The locale row

Add one entry to `LOCALES` in `scripts/translate/locales.ts`:

```ts
{
  tag: 'ko-KR',                 // BCP-47: a language AND a market
  language: 'Korean',
  market: 'South Korea',
  nameHints: '…translation guidance appended to the name instruction…',
  specialtyExamples: 'department-store food halls, import grocers, online',
  sections: KO_KR_SECTIONS,     // mirrors the vocabulary YAML slugs (§3)
}
```

`language`/`market`/`nameHints`/`specialtyExamples` are **English prompt
context**, not localized strings — they tune the generation prompt. The
`sections` array is the per-locale preferred errand vocabulary and must mirror
the vocabulary YAML slugs exactly. `CANONICAL_LOCALE` stays en-US.

## 3. Errand vocabulary — sections and stores

`l10n/vocabulary/<tag>.yaml` holds the store signage: the `sections` a shopper
walks and the three `stores` (primary / specialty / online).

### Sections are discovered, not invented

A single global aisle enum forces wrong fits across markets, so each locale's
sections come from an **open-coding pass over the data** (DESIGN: "Settled
during the translation workshop"). Run the four phases:

```
npx tsx scripts/translate/discover-errands.ts sample      # stratified sample
npx tsx scripts/translate/discover-errands.ts submit --model claude-opus-4-8
npx tsx scripts/translate/discover-errands.ts collect --batch-id msgbatch_…
npx tsx scripts/translate/discover-errands.ts aggregate   # cluster → proposal
```

Phase 1 asks the model for the **free-text sign a real supermarket hangs** —
no enum. Phase 4 clusters the answers into a proposed vocabulary. Then review
those clusters **against real store signage** (the launch locales were checked
against actual chain layouts — that pass caught Japan's 日配品, back-office
trade jargon no shopper ever sees, and replaced it with the real signs). Write
the frozen YAML:

```yaml
status: frozen
sections:
  - slug: meat                 # stable enum value modules ship
    label: 정육                # the sign text a Korean store hangs
    merges: [정육, 정육코너]    # discovery free-text that folds into this slug
  # …
stores:                        # the three Errand.store trips, localized
  primary: 마트
  specialty: 전문점
  online: 온라인
```

`slug` is the durable identifier; `label` is the display string; `merges`
records which discovery answers collapsed into the slug. The vocabulary is
**preferential, not enforced** — the generation schema constrains only
`store`; the model may coin a snake_case section when nothing fits, and
`strays.ts` surfaces those for review (§6).

## 4. Nutrient names — the deep one

This is the involved part. SR Legacy carries **149 distinct nutrients** (14
FDA-panel + 135 extra), and none of their names exist in any editable file —
they live only inside the vendored zip and render English in every locale
until you supply a table. You are producing `l10n/nutrients/<tag>.yaml`: a map
from the **stable USDA nutrient id** to a localized display name, covering
exactly those 149 ids.

### 4.1 Why id-keyed, and the two tiers

The nutrient *keys* (`nutrients.protein`, the extra rows' ids) are the
structural contract — identical in every locale — so they never translate.
Only the **display name** is per-locale, resolved through `labels.nutrients`
keyed by id. Two tiers, mirroring core/extra:

- **Panel (14 ids).** The en-US name is the **FDA Nutrition Facts panel
  wording** ("Total Fat", not USDA's "Total lipid (fat)"), defined in
  `LABEL_SET` (`src/toolkit/food.ts`). Other locales translate that concept.
- **Extra (135 ids).** The en-US name is the **USDA `nutrient.csv` name**
  verbatim. Both are produced by `canonicalNutrientNames`
  (`src/generator/nutrient-dictionary.ts`) — generated, never committed.

Get the exact target id list (and their USDA names, units, coverage) from the
generator:

```ts
import { buildNutrientDictionary } from '../src/generator/nutrient-dictionary.js';
import { loadDataset } from '../src/generator/load.js';
buildNutrientDictionary(loadDataset()); // 149 entries: { id, name, unit, foods, panel }
```

### 4.2 INFOODS tagnames — the provenance anchor

The names you source are scientific terms in a controlled vocabulary, not free
translation, and the anchor that keeps them honest across markets is the
**FAO/INFOODS tagname** — a language-independent component identifier from the
FAO/INFOODS *Tagnames* list (International Network of Food Data Systems). Each
USDA nutrient id maps to one tagname; the localized name is then the national
standard's term *for that tagname*. Examples of the tagname families you'll
meet:

| Nutrient kind | Tagname form | Examples |
|---|---|---|
| Proximates | mnemonic | `PROCNT` protein, `FAT` total fat, `CHOCDF` carbohydrate by difference, `WATER`, `ASH` |
| Energy | suffixed by unit | `ENERC_KCAL` (id 1008) vs `ENERC_KJ` (id 1062) — **same word, different tagname** |
| Minerals | element | `CA`, `FE`, `NA`, `ZN`, `SE`, `FD` (fluoride) |
| Fatty acids | `Fc_d` / `Fc_dNn` | `F16D0` (16:0), `F18D2` (18:2), `F22D6N3` (22:6 n-3, DHA) |
| Amino acids | `_G` suffix | `TRP_G` tryptophan, `LYS_G` lysine |
| Vitamers | mnemonic | `TOCPHA` α-tocopherol, `RETOL` retinol, `FOLDFE` folate DFE |

The tagname is provenance metadata: it ships nowhere (labels.js carries only
`id → name`), but it lives in the YAML `tagname:` field and the `basis` so a
reviewer can audit each name against the canonical component. Confirm tagnames
against the FAO/INFOODS list — **do not guess them**; several are
counterintuitive (see §4.6).

### 4.3 National food-composition standards — where the names come from

The localized display name is the term the market's own authority uses. For
the launch locales:

- **Japan (ja-JP):** 日本食品標準成分表2020年版（八訂） (MEXT Standard Tables of
  Food Composition) — including its 脂肪酸成分表編 (fatty acids) and
  アミノ酸成分表編 (amino acids) volumes.
- **China (zh-CN):** GB 28050-2011 (national nutrition-labelling standard) for
  the label-panel terms, 中国食物成分表 (China Food Composition Tables, Yang
  Yuexin ed.) for the full list, and WS/T 464-2015 (data-expression standard)
  for systematic names.

For a new market, find the equivalent national food-composition table and its
labelling regulation. Where the national table doesn't list a component (rare
vitamers, individual D2/D3, tocotrienols), fall back to the standard
**academic/chemical name** and say so in the `basis`.

### 4.4 The sourcing workflow (subagents, not the Batch API)

Source the names with **in-harness subagents**, not the paid Anthropic Batch
API — that path is reserved for the gated 7,793-food corpus
([[prefer-subagents-over-batch-api]]). 149 names is a bounded research task.
Translation-at-scale still needs the maintainer's explicit opt-in before you
spawn anything.

The shape that worked (one Workflow run, ~16 agents, ~10 min):

1. **Enumerate** the 149-id target (`buildNutrientDictionary`) and dump
   `{ id, usda_name, unit }` to JSON.
2. **Partition into nutrient families** so each agent owns a coherent slice and
   no agent carries too much. The split that balanced well:
   `proximates(15), sugars(6), minerals(11), vitamins(36), amino_acids(19),
   sterols(5), fatty_acids ×2 (29 + 28)`. Partition deterministically in code
   (first-match rules on the USDA name + id ranges) so the union is a guaranteed
   cover of all 149.
3. **Source** — one subagent per family, web-verifying against the national
   standard and the FAO/INFOODS tagname list. Force structured output:
   `{ id, tagname, ja, zh, ja_basis, zh_basis }` per nutrient. Tell it: concise
   *label* term (たんぱく質, not a description); keep fatty-acid carbon notation
   (16:0) intact; echo ids exactly; don't guess tagnames.
4. **Verify** — a second, adversarial subagent per family re-checks each name
   and tagname against the sources and corrects non-standard or transliterated
   terms. This pass earns its keep (see §4.6).
5. **Assemble and gate** — flatten, sort by id, assert coverage is exactly the
   149 dict ids with no blanks before writing anything.
6. **Convert to YAML** (§4.5) and run the tripwire (§4.7).

The one-off scripts for a run live in the gitignored `scripts/translate/out/`
(dictionary dump → workflow → YAML builder → verifier). They're throwaway by
design; this section is the durable recipe. If locales are added often, promote
the workflow into a committed generator under `scripts/translate/`.

### 4.5 The YAML format

```yaml
# <tag> nutrient names — id → localized display name for labels.nutrients.
status: sourced
sourced: 2026-06-13
nutrients:
  - id: 1003
    name: たんぱく質                                # what ships in labels.nutrients
    tagname: PROCNT                                 # INFOODS anchor (provenance)
    basis: MEXT 成分表2020（八訂）一般成分「たんぱく質」  # the decision log
  - id: 1258
    name: 飽和脂肪酸
    tagname: FASAT
    basis: MEXT 脂肪酸成分表編「飽和」; INFOODS FASAT
  # … all 149 ids …
```

The loader (`loadLocaleNutrientNames` in `scripts/translate/vocabulary.ts`)
reads `id` → `name`; `tagname` and `basis` are provenance the build ignores but
a reviewer relies on. An **empty list** (`nutrients: []`, `status: pending`)
means "not yet sourced" — the locale ships no nutrient names and the tripwire
skips it.

### 4.6 Gotchas (lessons from the launch run)

The reason this "sounds complicated" — the traps that actually came up:

- **Energy is listed twice.** id 1008 (kcal) and id 1062 (kJ) share a display
  word (エネルギー / 能量) but differ by tagname (`ENERC_KCAL` vs `ENERC_KJ`).
  Disambiguate by unit, not name.
- **Fatty acids are 95% systematic.** The carbon notation (`16:0`, `18:2 n-6`)
  is international; only the lipid-class head word and the trivial name
  localize (16:0 → 16:0（パルミチン酸）/ 棕榈酸 (16:0)). Don't translate the
  numbers.
- **The verify pass catches real errors.** On the launch run it fixed 16:1
  棕榈一烯酸 → 棕榈油酸 (the standard term) and standard-character slips
  (癸酸 vs the erroneous 葵酸). Budget for it — single-pass sourcing ships
  plausible-but-wrong terms.
- **Vitamers often aren't in the national table.** Individual D2/D3, the four
  tocotrienols, menaquinone-4 — national tables frequently list only the
  vitamin total. Fall back to the standard chemical name and note it.
- **"Added" nutrients have blank SR tagnames.** ids 1242 (Vitamin E, added) and
  1246 (B-12, added) carry no tagname in SR's `NUTR_DEF`; USDA/FDC convention
  is `VITE_ADD` / `VITB12_ADD`. Use those and say so.
- **IU forms are descriptive.** Vitamin A IU (1104) and Vitamin D IU (1110):
  national tables use μg RAE / μg, so the localized name is a descriptive
  "ビタミンA（IU）" / "维生素A（国际单位）", not a standard table row.
- **The folate family is four distinct components.** total (`FOL`), food
  (`FOLFD`), DFE (`FOLDFE`), and folic acid / synthetic (`FOLAC`) — each its own
  id and term; don't collapse them.
- **Units never localize.** kcal/kJ/g/mg/µg are international symbols and stay
  as-is; only names translate.

### 4.7 The tripwire

`tests/nutrient-labels.test.ts` enforces that a locale's nutrient table is
**either empty (pending) or covers exactly the 149 dataset ids** — a gap, a
stale id, or a blank name fails the build. This is the safety net that lets you
trust a sourced locale; run it before committing.

### 4.8 Making it simpler — what the launch run taught

The ja-JP + zh-CN run cost **≈270 tool calls across 16 subagents** (138 web
searches, 57 fetches, 37 shell commands incl. `curl`, `pip install pymupdf`,
`pdftotext`, and `iconv` encoding fights). Reading the trajectories back, most
of that was avoidable. If you (or an agent) source another locale, front-load
this so the agents only do the part that genuinely needs judgement — picking
the right national-standard term:

- **Resolve the INFOODS tagname once, offline — never per agent.** The biggest
  waste: all 16 agents independently chased the FAO/INFOODS tagname list, and
  several eventually rediscovered that the mapping is **already in USDA's own
  data** (`NUTR_DEF`, cited in their basis as "SR28 NUTR_DEF 319=RETOL"). The
  tagname is deterministic: USDA `nutrient.csv` carries `nutrient_nbr`, and the
  SR `NUTR_DEF` maps that number → tagname. Build `nutrientId → tagname` **once**
  (a committed reference, or a generator step) and pass it into the agents.
  Tagname is provenance, not translation — it should never be researched 16
  times. This alone removes most of the web searches.
- **Fetch each national standard once into a shared reference, not per agent.**
  16 agents separately `curl`-ed and `pdftotext`-ed the *same* MEXT and GB
  28050 PDFs, each re-fighting Windows codec issues (`tr -d` control chars,
  WINDOWS-1252→UTF-8). One prep step that pulls the standard's component list
  into a small UTF-8 text/CSV the agents read locally kills the duplicate
  fetches and the encoding pain entirely.
- **Pre-load the web tools.** `WebSearch`/`WebFetch` are deferred, so every
  agent spent its first turn on `ToolSearch("select:WebSearch,WebFetch")` — 16
  wasted round-trips. Note them in the prompt or hand the agent a fetch helper.
- **Put the gotchas (§4.6) in the prompt.** The agents *rediscovered* every one
  independently — energy kcal-vs-kJ tagnames, `VITE_ADD`/`VITB12_ADD` for the
  blank-tagname "added" nutrients, IU forms being descriptive, vitamers absent
  from national tables falling back to academic names. Stating these up front
  removes a whole class of dead-end searches.
- **Partition by difficulty, not just chemistry.** The real cost axis is "is
  this a label-standard term (one lookup) or an academic-only vitamer/trans-fat
  isomer (multi-source)?" The panel, minerals, proximates, amino acids, and
  common fatty acids are near-mechanical — carbon notation is universal, names
  are one-to-one. Give those a thin path (template + single confirmation) and
  reserve the deep research for the long tail (individual tocotrienols, MK-4,
  trans isomers, CLAs), which is where the 16th agent actually spent its time.
- **Each locale after the first is far cheaper.** ko-KR has the *same* 149 ids,
  the *same* tagnames, the *same* families, and the *same* gotchas — plus a
  ja/zh precedent column. Feed the agent the resolved tagname, the USDA name,
  and the existing ja/zh terms, and its only job is the Korean term from the
  Korean standard. Seed the new run from `l10n/nutrients/ja-JP.yaml` rather
  than from zero.

Net: a second locale should be a handful of agents doing one-lookup-per-name
against a pre-fetched standard, with the tagname and gotchas supplied — not 16
agents each rebuilding the reference scaffolding.

## 5. The food baseline

Names, aliases, errands, and notes for all 7,793 foods run through the
Anthropic **Message Batches API** (this is the one place the paid API is the
right tool — it's the full corpus, model-tiered for cost). Partition by tier,
submit in review-gated chunks, import to the per-food baseline:

```
npx tsx scripts/translate/router.ts --write                                   # opus-set / cheap-set
npx tsx scripts/translate/batch-claude.ts submit --input out/cheap-set.json --model claude-haiku-4-5
npx tsx scripts/translate/batch-claude.ts submit --input out/opus-set.json  --model claude-opus-4-8 --chunk 1 --of 3
npx tsx scripts/translate/batch-claude.ts collect --batch-id msgbatch_…
npx tsx scripts/translate/baseline.ts import out/<results>.jsonl              # JSONL wire → per-food YAML
```

JSONL is the wire format only; the stored baseline is one YAML file per food
under `l10n/baseline/<slug>.yaml`. Re-queue failed rows with `submit --retry`.

> **House rule:** paid batches beyond trivial scale require the maintainer's
> explicit go. Never fire the corpus run unprompted.

## 6. Review

- `npx tsx scripts/translate/strays.ts` — surfaces off-vocabulary section coins
  to correct or adopt into the vocabulary.
- `npx tsx scripts/translate/transliteration.ts` — sorts cheap-tier output by
  katakana/loanword density so a human can scan for literal-transliteration
  aliases the section audit can't see. It's a **noisy sort, not a classifier**.
- `l10n/ground-truth/<tag>.yaml` — **human-written**, durable across every
  regeneration, wins over any machine output, with `basis` as the
  glossary-decision log. Agents and batch jobs never write it: a model fixing
  machine output edits the stored baseline directly (same provenance class) and
  re-validates with `validateShape`.

## 7. Emit and validate

```
npm run build            # generates sr/ cores + manifest from the zip
npm run build:packages   # streams each locale tarball, labels.js baked in
npm test                 # the gates below must pass
```

The locale package ships `labels.js` (`{ sections, stores, nutrients }`) and
the `sr/<slug>{,.full}` views composing core leaves by reference. A consumer
renders with `localizeErrand` / `localizeNutrients` from the toolkit.

## 8. The tests that gate a locale

| Test | Gate |
|---|---|
| `tests/locales-vocab.test.ts` | the row's `sections` equal the vocabulary YAML slugs; `stores` label exactly primary/specialty/online |
| `tests/nutrient-labels.test.ts` | the nutrient table is empty (pending) or covers exactly the 149 dataset ids |
| `tests/errand-labels.test.ts` | `labels.js` carries sections + stores + nutrients; canonical reads generated names, others read their YAML |
| `tests/emit-l10n.test.ts` | the locale's modules emit per the leaf/view law; missing means missing (no English leak) |

When these are green for the new tag, the locale is structurally complete —
what remains is the reviewed quality of the sourced and baseline content.
